# DECISIONS

Принятые решения и отвергнутые альтернативы. Только факты, подтверждённые кодом или явным ТЗ сессии.

## Продукт

### D1. Только Telegram, без веба

**Решение:** весь UX в боте (reply + inline).  
**Почему:** ТЗ MVP явно запрещает админку.  
**Отвергнуто:** веб-дашборд, Mini App (можно позже, не сейчас).

### D2. Друзья без обязательного аккаунта

**Решение:** `Friend` — ручной контакт владельца; взаимодействие друга — только по deep-link долга (D24).  
**Почему:** учёт долгов для одного человека + опциональное подтверждение оплаты.  
**Отвергнуто:** обязательная регистрация друзей при создании контакта.

### D3. Долги только «мне должны»

**Решение:** `Debt` = друг должен пользователю.  
**Почему:** упрощение модели; в UI один раздел «Долги».  
**Отвергнуто (пока):** «я должен» как отдельный поток — **не реализовано**, требует нового ТЗ (см. TASKS T3.1).

### D4. Остаток копеек при equal split

**Решение:** остаток после округления → доля владельца; если владелец не в сплите → последний участник.  
**Почему:** сумма долей всегда равна total; правило задокументировано.  
**Отвергнуто:** случайный участник; банкнеровское «всем поровну с потерей копейки».

### D5. UI деления — сначала equal

**Решение:** в one-time FSM доступны «Поровну» и «Без деления»; percent/fixed живут в сервисе/тестах.  
**Почему:** быстрый MVP; калькулятор уже готов к расширению.  
**Отвергнуто:** полный UI всех режимов в первом релизе.

### D6. «Не списалось» → «Возникла проблема»

**Решение (обновлено D23):** вместо no-op — экран сценариев проблемы.  
**Исторически:** callback только уведомлял (no-op).  
**Отвергнуто:** автосдвиг даты без явного выбора пользователя.

## Техника

### D7. Decimal everywhere для денег

**Решение:** запрет float для сумм и курсов.  
**Почему:** требование ТЗ и корректность копеек.

### D8. Абстракция ExchangeRateProvider + CBR

**Решение:** интерфейс + `CbrExchangeRateProvider` + кэш `exchange_rates`.  
**Почему:** смена источника без переписывания converter.  
**Отвергнуто:** хардкод курса; тянуть курс «из будущего».

### D9. Один scheduler job

**Решение:** interval poll + `ReminderDelivery.unique_key`.  
**Почему:** ТЗ; проще идемпотентность, чем тысячи APScheduler jobs.  
**Отвергнуто:** job на каждую подписку.

### D10. Job шлёт сообщения сам

**Решение:** `process_reminders_job` вызывает `bot.send_message`.  
**Почему:** сервисы reminders готовят due-items без I/O TG; отправка — adapter в scheduler.  
**Компромисс:** формально нарушает «сервисы никогда не трогают TG», но boundary на job-файле.

### D11. Session middleware + commit после handler

**Решение:** одна AsyncSession на update, commit/rollback в middleware.  
**Почему:** простой DI для handlers.  
**Риск:** длинные внешние вызовы (CBR) внутри handler держат сессию — приемлемо для MVP.

### D12. create_all + Alembic

**Решение:** на старте `create_all`, схема также в Alembic revision.  
**Почему:** удобный локальный старт.  
**Отвергнуто (строго):** migrate-only без create_all — можно ужесточить позже (TASKS T2.4).

### D13. Memory FSM storage

**Решение:** `MemoryStorage`.  
**Почему:** zero infra.  
**Отвергнуто:** Redis до появления потребности в durable wizard.

### D14. UI Kit обязателен для новых экранов

**Решение:** `app/ui` + `.cursor/rules/telegram-ui-kit.mdc`.  
**Почему:** бот должен ощущаться как приложение; единые токены/кнопки/деньги.  
**Отвергнуто:** свободная разметка в каждом handler.

### D15. Статусы подписки не в БД

**Решение:** вычисляются в `resolve_subscription_status`.  
**Почему:** производные от `is_active` + `next_charge_date`.  
**Отвергнуто:** отдельная колонка status с риском рассинхрона.

### D16. Участники подписки отдельной таблицей

**Решение:** `subscription_participants` до первого charge.  
**Почему:** долги возникают при факте списания, не при создании подписки.  
**Отвергнуто:** хранить только на Transaction.

### D26. Подтверждённое списание уникально для периода подписки

**Дата:** 2026-07-21.

**Решение:** idempotency key — `subscription_id + transaction_date`; уникальность
обеспечивает индекс `uq_transactions_subscription_id_transaction_date`.
Transaction, splits, debts и новая дата подписки создаются атомарно.

**Почему:** повторный или параллельный callback не должен создавать двойное
списание, долги или повторно сдвигать дату.

**Подробности:** `docs/CHARGE_IDEMPOTENCY.md`.

## Отображение

### D17. Без «ЦБ» и tech jargon в UI (с оговоркой)

**Решение:** в карточках подписок и обычных экранах — «официальный курс», «≈ будет рассчитано автоматически», без «ЦБ».  
**Почему:** потребительский тон (UX-аудит).  
**Оговорка:** на подтверждении разового платежа допускается «Курс ЦБ» (D25); ask-first charge superseded by D23.

### D20. Экран фактического списания (исторический)

**Дата:** 2026-07-14.  
**Статус:** **superseded by D23**.  
Исторически: ask-first «В РУБЛЯХ» / «По курсу ЦБ» после «Списалось» для иновалюты.

### D21. Навигация и команды важнее FSM

**Дата:** 2026-07-14.  
**Решение:** `NotNavigationOrCommand` + `state.clear()` на командах и Nav; порядок роутеров cancel → start → settings → …  
**Отвергнуто:** middleware очистки state до handlers.

### D22. Редактируемая карточка списания

**Дата:** 2026-07-14.  
**Решение:** карточка `Transaction` с правками суммы/даты/курса/долгов/отмены/удаления.  
Отмена откатывает `next_charge_date`; удаление только чистит историю.

### D23. Подтверждение списания без ввода суммы (Product Review)

**Дата:** 2026-07-14.  
**Контекст:** ask-first (D20) лишний для MVP регулярных подписок.  
**Решение:** «Подтвердить списание» → сразу save + следующая дата; «Возникла проблема» → 5 сценариев (без Перенести/Пропустить).  
**Отвергнуто:** обязательный ввод фактических ₽ на happy path.

### D24. Deep-link двухэтапного подтверждения долга

**Дата:** 2026-07-14.  
**Решение:** `share_token` + `/start debt_<token>`; статусы active → needs_review → paid; друг не закрывает долг сам; remind 24ч.  
**Отвергнуто:** обязательный telegram_id у Friend; закрытие долга другом в один клик.

### D25. Разовый платёж: новый порядок мастера и всегда CBR

**Дата:** 2026-07-14.  
**Решение:** name → currency → amount → date → card → participants → split → confirm; категория OTHER; курс официальный.  
**Отвергнуто:** ранний выбор категории; шаг «как посчитать рубли» в создании.

### D18. Reply = навигация, Inline = действия

**Решение:** `Nav` / `Action` в UI Kit.

### D19. Кнопки на карточке, не отдельным «Действия:»

**Решение:** `edit_text(..., reply_markup=...)`.

## Отложенные сознательно (не забыть)


| Тема                      | Почему отложено                |
| ------------------------- | ------------------------------ |
| «Я должен»                | Нет модели/экрана              |
| Банки / SMS               | Вне MVP                        |
| Полный percent UI         | Calculator готов, FSM нет      |
| Edit one-time после save  | Кнопка есть, FSM нет (T1.6)    |
