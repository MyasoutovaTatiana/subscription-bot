# Идемпотентность подтверждения списания

## Модель

Одно подтверждённое списание определяется парой
`subscription_id + transaction_date`. Повторный callback для той же пары
возвращает уже созданную транзакцию и не создаёт новые splits или debts и не
сдвигает `next_charge_date` повторно.

Уникальность защищена индексом
`uq_transactions_subscription_id_transaction_date`. Для разовых платежей
`subscription_id` равен `NULL`, поэтому индекс не запрещает несколько платежей
на одну дату.

Создание Transaction, splits, debts и обновление `next_charge_date` выполняются
в одной savepoint-транзакции. Ошибка на любом из этих этапов откатывает всю
операцию; после отката подтверждение можно повторить.

## Проверка перед миграцией

Перед созданием индекса выполнить запрос:

```sql
SELECT subscription_id, transaction_date, COUNT(*) AS count
FROM transactions
WHERE subscription_id IS NOT NULL
GROUP BY subscription_id, transaction_date
HAVING COUNT(*) > 1;
```

Миграцию можно продолжать только при пустом результате. Сама Alembic-миграция
также останавливается, если находит дубликаты.

## Резервная копия и rollback

Перед production-миграцией создать проверенную резервную копию базы вне
рабочего файла. Для SQLite корректный способ — SQLite Backup API; после копии
обязательно выполнить `PRAGMA integrity_check`.

Если миграция не завершилась или проверка после неё не прошла:

1. Остановить бот, чтобы исключить запись в базу.
2. Не повторять `stamp` или `upgrade`, пока не проверена текущая ревизия.
3. Проверить `alembic_version`, наличие индекса и `PRAGMA integrity_check`.
4. Если индекс создан, а данные целы, выполнить
   `python -m alembic downgrade a1b2c3d4e5f6`.
5. Если база повреждена или состояние неоднозначно, сохранить проблемный файл и
   восстановить рабочую БД из проверенной pre-migration копии.
6. Повторно проверить целостность и только затем запустить бот.
