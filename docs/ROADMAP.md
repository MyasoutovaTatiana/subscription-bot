# ROADMAP

Актуальная версия: 2026-07-16.

## Продуктовый контур

Проект развивает только:

- регулярные подписки и напоминания о списаниях;
- совместные подписки и регулярный сбор денег;
- совместные разовые расходы как источник долгов;
- долги в обе стороны: «Мне должны» и «Я должна».

В roadmap не входят личный бюджет, ежедневный учёт трат и доходов, универсальная финансовая аналитика и банковские интеграции ближайших релизов.

## Целевая архитектура

```text
Telegram
  ↓
Telegram Bot / aiogram
  ↓
Application services / backend API
  ↓
PostgreSQL
  ↑
Admin API
  ↑
Закрытая web-админка
```

Production-контур должен работать на российской инфраструктуре. Railway и SQLite считаются временными и отключаются только после проверенной миграции данных.

## Критический путь

1. Зафиксировать текущее состояние и резервные копии.
2. Закрыть Security Gate.
3. Выбрать и развернуть российскую инфраструктуру.
4. Перейти с Railway SQLite на PostgreSQL.
5. Запустить защищённую read-only админку.
6. Завершить подписки и долги в обе стороны.
7. Подключить друзей и регулярные сборы.
8. Повысить надёжность.
9. Позднее добавить OCR и Mini App для общих счетов.

## Релизы

### v0.1 — Architecture & Planning

- выбрать российское облако;
- зафиксировать архитектуру bot / API / admin / database / infrastructure;
- спроектировать PostgreSQL и админку;
- завести ADR по ключевым решениям;
- создать GitHub roadmap и backlog.

### v0.2 — Security Gate

- провести ownership-аудит всех пользовательских сущностей;
- закрыть IDOR через callback data, FSM и deep-link;
- проверить атомарность создания платежей, списаний и долгов;
- провести аудит секретов и Git history;
- добавить security regression tests и CI;
- проверить сценарии на двух Telegram-аккаунтах.

Новые крупные продуктовые функции не начинаются до закрытия P0-задач этого релиза.

### v0.3 — Russian Infrastructure

- сравнить российских провайдеров и зафиксировать выбор ADR;
- развернуть staging в российском облаке;
- создать приватную сеть;
- вынести секреты в Secret Manager;
- настроить мониторинг, логи и healthcheck;
- разделить staging и production.

### v0.4 — PostgreSQL Migration

- провести ревизию схемы;
- сделать Alembic единственным способом изменения production-схемы;
- создать роли БД с минимальными правами;
- перенести данные из Railway SQLite;
- настроить бэкапы и проверить восстановление;
- выполнить финальное переключение и отказаться от Railway.

### v0.5 — Admin MVP

- реализовать защищённую авторизацию администратора;
- создать read-only разделы пользователей, подписок, платежей и долгов;
- добавить поиск и фильтры;
- показывать ошибки и состояние scheduler;
- журналировать входы и административные действия;
- не давать прямой SQL-доступ к production-базе.

### v1.0 — Core Product

- завершить мастер создания подписки;
- добавить участников и способы деления подписки;
- создавать долги после подтверждённого списания;
- сделать однозначными итоги за период;
- реализовать долги «Мне должны» и «Я должна»;
- добавить плательщика и расширенное деление совместных платежей;
- реализовать полное редактирование платежа и пересчёт незакрытых долгов.

### v1.1 — Connected Friends & Collections

- приглашать друга по персональной ссылке;
- связывать ручную запись Friend с Telegram ID;
- отправлять уведомления должнику;
- поддержать «Я оплатил» и двухэтапное подтверждение;
- создать регулярный цикл сборов по совместным подпискам;
- показывать историю взаиморасчётов.

### v1.2 — Reliability

- перенести FSM из MemoryStorage в постоянное хранилище;
- реализовать экспорт и безопасное удаление данных;
- оптимизировать PostgreSQL-запросы;
- провести нагрузочное тестирование scheduler;
- регулярно проверять disaster recovery.

### v2.0 — Group Bills

- исследовать OCR чеков;
- распознавать и редактировать позиции;
- создать Mini App для распределения общего счёта;
- поддержать сценарий Telegram-группы;
- создавать долги только после подтверждения итогового распределения.

## Приоритеты

- P0 — безопасность, сохранность данных, инфраструктура и миграция;
- P1 — основное продуктовое ядро подписок и долгов;
- P2 — надёжность и эксплуатация;
- P3 — OCR, Mini App и дальнейшее развитие.

## Definition of Done

Для каждой задачи:

1. Зафиксированы проблема, сценарий и критерии готовности.
2. Код использует ownership checks и минимальные права доступа.
3. При необходимости добавлена миграция Alembic.
4. Нет секретов и чувствительных данных в коде и логах.
5. Добавлены unit / integration / security tests.
6. `pytest` и CI проходят успешно.
7. Проведён ручной smoke-test для пользовательского сценария.
8. Обновлены STATUS, PRODUCT_RULES, DECISIONS или ROADMAP, если изменились правила.
